C. The Last Harness You'll Ever Build
==========================================
AI agent가 여러 도메인에 적용될때 painstaking, expert-driven 하네스 엔지니어링이 필요한데 2단계 루프를 통해 이를 자동화하겠다

Inner Loop: Harness Evolution Loop 가 수정하는것

시스템 프롬프트와 작업 프롬프트
에이전트의 역할(identity)과 constraint을 정의하는 시스템 수준 지시사항, 그리고 목표·성공 기준·인컨텍스트 예제를 정의하는 작업 수준 프롬프트.

도구(tools), 스킬(skills), 그리고 그 설명
에이전트가 환경과 상호작용하기 위해 호출할 수 있는 기능들.
(예: 파일 수정, 셸 실행, UI 조작, 웹 검색, MCP 서버 등)

번들된 인프라(Bundled infrastructure)
에이전트에게 제공되는 실행 환경.
(파일 시스템, 샌드박스, 브라우저, 관측/로깅 스택 등)
<img width="849" height="597" alt="image" src="https://github.com/user-attachments/assets/601045f4-3d1e-4a6b-9070-d93be4add0be" />
오케스트레이션 로직(Orchestration logic)
에이전트의 상호작용 루프를 구성하는 제어 흐름.
(서브에이전트 생성, 핸드오프, 모델 라우팅, 피드백 루프, Ralph Loop 같은 continuation 패턴 등)

훅(hooks)과 미들웨어(middleware)
모델 주변에 삽입되는 결정론적 실행 보장 장치.
(컨텍스트 압축, continuation, lint 검사, 검증 루프 등)

모델 설정(Model configurations)
어떤 모델을 사용할지, 추론 파라미터(temperature, 샘플링 전략, 토큰 제한), 그리고 어떤 하위 작업을 어떤 모델이 처리할지 결정하는 모델 라우팅 규칙 등을 포함한다.
<img width="1346" height="499" alt="image" src="https://github.com/user-attachments/assets/6e20640b-43cd-4796-9de9-74ac208b8337" />

수정과정 
1. Agent Worker (W_H)
하네스(H)를 장착한 상태로 task 실행
execute(task) → trace

2. Evaluator (V)
trace를 보고 진단 + 점수 평가
evaluate(trace, task) → (report, score)

3. Evolution (E)
이전 히스토리를 기반으로 harness 수정 및 개선
evolve(history, best_harness) → new_harness

Outer Loop: Meta-Evolution Loop 가 수정하는것

Evolution Agent 프롬프트, Evaluator 프롬프트, evaluator 점수 함수 설계, loop 하이퍼파라미터 등 inner loop를 구성하는 모든 것.

D. Life-Harness
==========================================
C. Life Harness
4개 레이어가 서로 다른 시점에 끼어듦
trajectory(실행 기록)들을 바탕으로 발전하며,
반복적으로 발생하는 interaction 실패를 재사용 가능한 skill과 같은 intervention으로 변환

시작하기 앞서 failed trajectories 를 수집 생 LLM으로 
ex) AlFWorld에서 Qwen3 4B로 뽑은 traj를 Codex를 통해 실패유형 4개중 한개로 매핑 

1. 행동실패 
의도는 맞는데 syntax 같은것 때문에 실행이 안됨
tool_call 대신 그냥 텍스트로 함수 적음
take_action({...}) 이렇게 채팅에 써버림

JSON 형식 깨짐
함수 이름 틀림
required argument 빠짐
argument 타입 틀림
<img width="1233" height="483" alt="image" src="https://github.com/user-attachments/assets/fd699e56-63fa-486b-a25c-355808ff98b2" />

2. 환경 Contract Mismatches
여기서는 tool call 자체는 실행 가능함
근데 “환경 규칙”을 잘못 이해함
예: ["apple", "banana"] 해야되는데 The answer is apple and banana

finish tool 쓰면 안 되는데 사용
answer 제출 전에 중간 tool 먼저 써야 하는데 생략
search tool을 이상하게 사용
형식은 JSON 맞는데 의미가 틀림

3. Trajectory Degeneration
이건 개별 action은 실행 잘 됨, 근데 긴 실행 과정에서 전략이 망가짐
nothing happens 가 계속되는데도 수정 X

같은 행동 반복
같은 페이지 계속 방문
search만 계속 반복
look / inventory 무한 반복
상태 변화 없는데 계속 retry
budget exhausted

4. Residual Reasoning Failures
위의 3개에 안속하면 이제 다 여기, 대부분 모델의 사고(reasoning)가 틀림

계산 실수
SQL 논리 틀림
잘못된 객체 선택
retrieval 실패
filtering 조건 오류
최종 값 선택 오류

1. Environment Contract Layer = “환경 규칙 설명서”

에이전트 시작 전에:
“여기선 이런 규칙으로 행동해야 함”
을 추가로 알려줌.

상품권 최대 3개
왕복 항공편은 같이 제출
구매 전 옵션 선택 필수

2. Procedural Skill Layer = “공략집”

과거 성공 trajectory에서 자주 통하는 전략을 저장해놨다가 현재 task와 비슷한 스킬 하나를 꺼내줌.

DB 수정 전에 먼저 preview SELECT 해라
grep -c 결과는 파일별 count라 합산 필요
WebShop에선 petite/tall도 옵션일 수 있음

3. Action Realization Layer= “실행 직전 검사기”

모델이 action 생성하면 바로 실행하지 않고:
“이거 환경에서 실제 가능한 행동인가?”
를 체크함.

좌석 없는 항공편 예약 → 차단
옵션 선택 없이 buy now → 차단
SQL 문법 살짝 틀림 → 자동 보정

4. Trajectory Regulation Layer= “루프 감시 시스템”

실행 후 trajectory 전체를 보면서:
“지금 같은 행동 반복 중인가?”
를 감시함.

search만 계속 반복
같은 페이지 계속 방문
look/inventory 무한 반복

Evolution
하네스를 사람이 일일이 만든 게 아니라, 실패 trajectory를 보고 계속 자동 개선

고정된(frozen) 모델을 task들에 계속 실행함
→ 에이전트 행동 기록(trajectory) 수집
Codex가 그 trajectory를 읽음
→ 어디서 반복적으로 실패하는지 분석
실패 패턴 발견하면
→ 하네스 레이어 업데이트 제안
