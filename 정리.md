A. Agentic Harness Engineering 2026/04
==========================================

1. 수백만 토큰의 raw trace에서, 막 multi step에 skill.md 쓰고 tool call, subagent, MCP 이렇게 쓰면 "어떤 컴포넌트가 왜 실패했는지" 알기 어려움

2. 컴포넌트 간 강한 결합 → 편집이 불안정
기존 하네스 프레임워크는 prompt/tool/middleware가 얽혀있어서, 하나 건드리면 다른 게 망가짐. 편집의 action space 자체가 불명확

C. The Last Harness You'll Ever Build
==========================================
AI agent가 여러 도메인에 적용될때 painstaking, expert-driven 하네스 엔지니어링이 필요한데 2단계 루프를 통해 이를 자동화하겠다

Inner Loop: Harness Evolution Loop 가 수정하는것

시스템 프롬프트와 작업 프롬프트
에이전트의 역할(identity)과 constraint을 정의하는 시스템 수준 지시사항, 그리고 목표·성공 기준·인컨텍스트 예제를 정의하는 작업 수준 프롬프트.

도구(tools), 스킬(skills), 그리고 그 설명
에이전트가 환경과 상호작용하기 위해 호출할 수 있는 기능들.
(예: 파일 수정, 셸 실행, UI 조작, 웹 검색, MCP 서버 등)

번들된 인프라(Bundled infrastructure)
에이전트에게 제공되는 실행 환경.
(파일 시스템, 샌드박스, 브라우저, 관측/로깅 스택 등)
<img width="849" height="597" alt="image" src="https://github.com/user-attachments/assets/601045f4-3d1e-4a6b-9070-d93be4add0be" />
오케스트레이션 로직(Orchestration logic)
에이전트의 상호작용 루프를 구성하는 제어 흐름.
(서브에이전트 생성, 핸드오프, 모델 라우팅, 피드백 루프, Ralph Loop 같은 continuation 패턴 등)

훅(hooks)과 미들웨어(middleware)
모델 주변에 삽입되는 결정론적 실행 보장 장치.
(컨텍스트 압축, continuation, lint 검사, 검증 루프 등)

모델 설정(Model configurations)
어떤 모델을 사용할지, 추론 파라미터(temperature, 샘플링 전략, 토큰 제한), 그리고 어떤 하위 작업을 어떤 모델이 처리할지 결정하는 모델 라우팅 규칙 등을 포함한다.

수정과정 
1. Agent Worker (W_H)
하네스(H)를 장착한 상태로 task 실행
execute(task) → trace

2. Evaluator (V)
trace를 보고 진단 + 점수 평가
evaluate(trace, task) → (report, score)

3. Evolution (E)
이전 히스토리를 기반으로 harness 수정 및 개선
evolve(history, best_harness) → new_harness

Outer Loop: Meta-Evolution Loop 가 수정하는것

Evolution Agent 프롬프트, Evaluator 프롬프트, evaluator 점수 함수 설계, loop 하이퍼파라미터 등 inner loop를 구성하는 모든 것.
