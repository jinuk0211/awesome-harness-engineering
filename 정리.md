C. The Last Harness You'll Ever Build
==========================================
AI agent가 여러 도메인에 적용될때 painstaking, expert-driven 하네스 엔지니어링이 필요한데 2단계 루프를 통해 이를 자동화하겠다

Inner Loop: Harness Evolution Loop 가 수정하는것

시스템 프롬프트와 작업 프롬프트
에이전트의 역할(identity)과 constraint을 정의하는 시스템 수준 지시사항, 그리고 목표·성공 기준·인컨텍스트 예제를 정의하는 작업 수준 프롬프트.

도구(tools), 스킬(skills), 그리고 그 설명
에이전트가 환경과 상호작용하기 위해 호출할 수 있는 기능들.
(예: 파일 수정, 셸 실행, UI 조작, 웹 검색, MCP 서버 등)

번들된 인프라(Bundled infrastructure)
에이전트에게 제공되는 실행 환경.
(파일 시스템, 샌드박스, 브라우저, 관측/로깅 스택 등)
<img width="849" height="597" alt="image" src="https://github.com/user-attachments/assets/601045f4-3d1e-4a6b-9070-d93be4add0be" />
오케스트레이션 로직(Orchestration logic)
에이전트의 상호작용 루프를 구성하는 제어 흐름.
(서브에이전트 생성, 핸드오프, 모델 라우팅, 피드백 루프, Ralph Loop 같은 continuation 패턴 등)

훅(hooks)과 미들웨어(middleware)
모델 주변에 삽입되는 결정론적 실행 보장 장치.
(컨텍스트 압축, continuation, lint 검사, 검증 루프 등)

모델 설정(Model configurations)
어떤 모델을 사용할지, 추론 파라미터(temperature, 샘플링 전략, 토큰 제한), 그리고 어떤 하위 작업을 어떤 모델이 처리할지 결정하는 모델 라우팅 규칙 등을 포함한다.
<img width="1346" height="499" alt="image" src="https://github.com/user-attachments/assets/6e20640b-43cd-4796-9de9-74ac208b8337" />

수정과정 
1. Agent Worker (W_H)
하네스(H)를 장착한 상태로 task 실행
execute(task) → trace

2. Evaluator (V)
trace를 보고 진단 + 점수 평가
evaluate(trace, task) → (report, score)

3. Evolution (E)
이전 히스토리를 기반으로 harness 수정 및 개선
evolve(history, best_harness) → new_harness

Outer Loop: Meta-Evolution Loop 가 수정하는것

Evolution Agent 프롬프트, Evaluator 프롬프트, evaluator 점수 함수 설계, loop 하이퍼파라미터 등 inner loop를 구성하는 모든 것.

D. Life-Harness
==========================================
4개 레이어가 서로 다른 시점에 끼어듦
trajectory(실행 기록)들을 바탕으로 발전하며,

F. Ctx2Skill
===================
책, 논문, 깃허브 리포와 같은 컨텍스트를 스킬로 바꿈

Challenger 프롬프트
Task 설계 + 루브릭 설계 규칙 두 파트로 나뉨
핵심은 루브릭: "단순 검색으로 풀리면 안 됨", "이진 검증 가능해야 함"
런 루브릭이 있어야 Judge가 외부 정답 없이 채점 가능 → external 피드백 없어도 루프 가능 << 문제 해결하는 핵심
-> 주어진 컨텍스트에 대해 Task + 루브릭 생성

Judge 프롬프트
3단계 강제: 분석 → 검증 → 자기반성
루브릭 단 하나라도 fail이면 전체 0점 (all-or-nothing)
시스템 프롬프트 없이 유저 메시지만 → 다른 에이전트들과 달리 스킬 진화에 관여 안 함, 중립 유지
-> Task 생성된거에 루브릭 기준으로 점수매김

실패시)
Reasoner Proposer 프롬프트

Judge 실패 유형을 6가지로 분류: 내용공백 / 형식오류 / 제약위반 / 추론오류 / 태스크오해 / 시스템프롬프트 미준수

Reasoner Generator 프롬프트
Reasoner Proposer이 생성한 diagnosis을 실제 SKILL.md로 구체화
구조 4파트 강제: 사전 체크리스트 / 응답 절차 / 자기검증 / 함정
YAML 프론트매터 두 필드만 (skill_name, skill_description)

## Ctx2Skill 구현 방식 딥다이브

---

### 1. 전체 파이프라인 (알고리즘 흐름)

```
Input: Context C, iterations N=5, tasks per iter M=5

초기화:
  S_R^0 = ∅  (Reasoner 스킬셋, 빈 상태로 시작)
  S_C^0 = ∅  (Challenger 스킬셋)
  Q_hard = []  (하드 프로브셋)
  Q_easy = []  (이지 프로브셋)

for i = 1..N:
  ① Challenger(C, S_C^{i-1}) → {(t_m, R_m)} × M개 태스크+루브릭
  ② Reasoner(C, S_R^{i-1}, t_m) → a_m  (M개 답변 생성)
  ③ Judge(t_m, R_m, a_m) → y_m ∈ {0,1}  (루브릭 전체 pass면 1)
  
  ④ 실패케이스 F_i = {m : y_m = 0}
     → Reasoner Proposer(F_i, S_R^{i-1}) → 진단 d_R
     → Reasoner Generator(d_R, S_R^{i-1}) → S_R^i
  
  ⑤ 성공케이스 P_i = {m : y_m = 1}
     → Challenger Proposer(P_i, S_C^{i-1}) → 진단 d_C
     → Challenger Generator(d_C, S_C^{i-1}) → S_C^i
  
  ⑥ 프로브 누적:
     Q_hard += argmin_{m∈F_i}(루브릭 통과율)  # 가장 어렵게 실패한 태스크
     Q_easy += argmin_{m∈P_i}(루브릭 수)      # 가장 쉽게 성공한 태스크

// Cross-Time Replay
for i = 1..N:
  ρ_h(i) = (Q_hard에서 S_R^i로 맞힌수 + 1) / (|Q_hard| + 1)  # Laplace smoothing
  ρ_e(i) = (Q_easy에서 S_R^i로 맞힌수 + 1) / (|Q_easy| + 1)

i* = argmax_i [ ρ_h(i) × ρ_e(i) ]
return S_R^{i*}
```

### 2. 각 에이전트의 실제 프롬프트 구조

#### Challenger
```
System: 너는 컨텍스트 전문가야
Input:
  - Context C (전체 문서)
  - S_C^{i-1} (Challenger 스킬셋)
  
Output: M개의 태스크, 각각에 binary 루브릭 리스트
  예시 루브릭:
    - "답변에 3단계 절차가 명시되어 있는가?"
    - "컨텍스트의 조항 3.2를 인용하는가?"
    - "수치가 소수점 2자리인가?"
  
핵심 제약: "컨텍스트 표면만 paraphrase하면 pass되는 태스크 금지"
         "룰을 귀납적으로 이해해야 풀 수 있어야 함"
```

#### Judge (엄격한 all-or-nothing)
```
3단계 채점:
  Step 1: 각 루브릭의 요구사항 분석
  Step 2: 루브릭별 pass/fail 개별 판정
  Step 3: 자기 반성 (놓친 게 있는지)

y_m = 1  오직 모든 루브릭이 pass일 때만
→ 루브릭 11개 중 10개 맞아도 y_m = 0
```

#### Proposer (진단 전문가)
```
Reasoner Proposer:
  Input: 실패 케이스들 F_i + 현재 S_R^{i-1}
  출력 형식 (structured):
    - action: "add" | "merge"
    - target_skill_name: str
    - description: str (무슨 지식이 빠졌는가)
    - justification: str (왜 이 케이스들이 그 증거인가)
  
  핵심: 케이스별 진단 X → 여러 케이스의 공통 패턴 합성
  예: "실패 4건이 모두 예외조항 처리 미흡 → add skill: 예외처리 프로토콜"
```

#### Generator (스킬 구체화)
```
Input: Proposer의 진단 + 현재 스킬셋
Output: 완전한 새 SKILL.md (기존 스킬 보존 + 신규/병합 반영)

규칙:
  - 관련없는 기존 스킬은 그대로 유지
  - add → 새 섹션 추가
  - merge → 기존 섹션에 통합
```

### 3. Cross-Time Replay 구현 디테일

**왜 마지막 iteration 스킬을 쓰면 안 되냐?**

실험 결과 GPT-4.1 기준:
| Iter | Solving Rate |
|------|-------------|
| 1 | 15.9% |
| 2 | 15.5% |
| 3 | 15.3% |
| 4 | 14.9% |
| 5 | 14.7% |
| **Cross-Time Replay** | **16.5%** |

→ iter 늘어날수록 오히려 나빠짐. Challenger가 점점 극단적 태스크만 내니까 Reasoner 스킬이 그것에 과적합됨

**선택 로직**:
```python
# Laplace smoothing으로 빈 프로브셋 edge case 처리
rho_hard = (hard_solved + 1) / (len(Q_hard) + 1)
rho_easy = (easy_solved + 1) / (len(Q_easy) + 1)

score = rho_hard * rho_easy  # 곱셈 중요!
# 왜 덧셈 아니고 곱셈?
# → 덧셈이면 hard=0%, easy=100%도 score=1.0
# → 곱셈이면 한쪽 0%면 score=0 → 극단적 스킬 자동 제외
```

**프로브셋 구성 방식**:
- `Q_hard`: 매 iter마다 "루브릭 통과율 최저인 실패 태스크" 1개 추가 → 5 iter면 최대 5개
- `Q_easy`: 매 iter마다 "루브릭 수 최소인 성공 태스크" 1개 추가

즉 N=5면 Cross-Time Replay에서 각 후보 스킬셋을 최대 10개 태스크로 재평가하는 것

### 6. 아키텍처 관점 핵심 설계 결정

**왜 Proposer/Generator 분리했나?**
- 합치면 -0.5% 하락
- 진단(what & why)과 구체화(how)를 분리해야 품질 높아짐
- LLM이 한 번에 "문제 분석 + 스킬 작성" 하면 진단이 얕아짐

**왜 두 사이드 스킬을 서로 안 보여주나?**
- `S_R`과 `S_C`는 서로 격리됨
- 만약 Reasoner가 Challenger 스킬을 보면 → 시험 답지 보는 꼴
- 엄격한 adversarial 구조 유지가 핵심

**왜 both sides를 업데이트하나 (Loser-Only 대비 +0.5%)?**
- Challenger도 성공 케이스 보고 "다음엔 이 방향 피해야지" 학습
- 한쪽만 업데이트하면 Challenger가 진화 안 해서 adversarial pressure 약해짐

I. Harness Updating Is Not Harness Benefit: Disentangling Evolution Capabilities in Self-Evolving LLM Agents
==========================================
하네스를 자동으로 진화시킨다는 건 좋은데, 그 이득이 대체 누구 덕인지를 아무도 안 따졌다. 이 논문은 "하네스를 잘 **써주는** 능력"과 "써준 하네스로 잘 **이득 보는** 능력"을 분리해서 측정한 분석 논문이다. (방법 제안이 아니라 측정/해부 논문)

깔고 가는 전제:
LLM agent = 고정된 모델 + 바깥의 편집 가능한 하네스(프롬프트·skill·메모리·툴).
하네스 self-evolution = 모델 weight는 안 건드리고, 실행 기록(trajectory, 실패, 피드백)만 보고 하네스를 계속 고쳐 쓰는 것.

여기서 두 역할이 나뉜다:
- evolver: 실행 기록 읽고 하네스 업데이트를 **써주는** 놈
- solver(task-solving agent): 업데이트된 하네스를 **써먹는** 놈

기존 평가의 한계:
"이 self-evolution 방법이 성능 올렸나?"라는 end-to-end 질문만 던짐.
→ 이득이 evolver가 잘 써줘서인지, solver가 잘 써먹어서인지 구분 불가.

`<img — fig_overview_harness_self_evolution.pdf : 하네스 진화 루프 개요>`

==========================================
그래서 두 능력을 따로 정의:

1. harness-updating (evolver 역할) — 쓸모 있는 하네스 업데이트를 만들어내는 능력
2. harness-benefit (solver 역할) — 업데이트된 하네스로부터 이득 보는 능력

그리고 이 둘을 base capability(하네스 없이 그냥 task 푸는 맨몸 실력)와 대조한다.

비유하면: **좋은 코치인 것 ≠ 코칭을 잘 흡수하는 선수인 것 ≠ 그냥 운동 잘하는 것.** 셋이 다 별개라는 게 핵심 주장.

셋업:
7개 LLM(Claude Opus/Sonnet 4.6, Haiku 4.5, Qwen3-235B/32B, Qwen3.5-9B, GPT-OSS-120B) × 3개 벤치(SWE-bench Verified, MCP-Atlas, SkillsBench).
agent × evolver 다 짝지어서 매트릭스로 돌림. 같은 task stream, 같은 진화 예산, LLM만 바꿈.

`<img — fig_intro_findings_v2.pdf : 두 발견 요약 (flat vs non-monotonic)>`

==========================================
발견 1 — harness-updating은 base 실력에 대해 평평(flat)하다

solver 고정하고 evolver만 바꿔봄. 근데 capability tier가 천차만별인데 만들어내는 업데이트의 효과가 놀랄 만큼 비슷했다.

- evolver 간 best-worst 격차가 어떤 벤치에서도 최대 3.1pp
- 모든 벤치를 다 이기는 evolver는 없음. 예: Qwen3-235B는 SWE 1등(8.2pp)인데 MCP 꼴등(0.6pp)
- **모델 크기 무관**: 제일 작은 9B(Qwen3.5-9B)가 SkillsBench에서 1등(3.8pp). Opus 4.6(2.3pp), Qwen3-235B(1.5pp)보다 높음

왜 비슷한가 (flink-query 케이스):
solver를 Opus 4.6로 고정하고 skill만 바꿔봄.
- skill 없음 → FINISH-event 필터 빼먹고 실패 (0.67)
- 9B가 써준 skill → 성공 (1.0)
- Opus가 써준 skill → 성공 (1.0)
두 skill을 까보니 "procedurally isomorphic" — 같은 절차 순서를 처방하고 표현 장황함·구현 디테일만 다름. 즉 9B든 frontier든 **도달하는 절차적 내용은 같다.**

`<img — evolver_side_delta_bars.pdf + evolver_side_case.pdf : evolver별 gain 막대 / 9B vs Opus skill 비교>`

==========================================
발견 1의 따름정리 — post-evolution 점수는 evolver가 아니라 solver의 base 실력이 지배한다

같은 solver에서 evolver 7개를 다 바꿔봐도:
- within-agent spread(evolver 갈아끼운 변동폭) ≤ 5.1pp
- 근데 agent끼리 base 격차(Opus vs Qwen3-235B)는 36.0pp

극단 실험:
약한 agent에 **최고** evolver 붙이기 vs 강한 agent에 **최악** evolver 붙이기
→ 그래도 강한 agent가 18.6~35.2pp 이김.

결론: 병목은 evolver 쪽이 아니라 solver 쪽이다.

`<img — evolver_side_agent_dominance_mcp_v2.pdf : agent별 점수, evolver 변동은 작고 agent 간 격차는 큼>`

==========================================
발견 2 — harness-benefit은 base 실력에 대해 비단조(non-monotonic)다

이번엔 evolver 고정하고 solver만 바꿈. 이득이 base 실력 따라 단조증가하지 **않는다.**
약한 모델은 거의 못 먹고 → **중간 모델이 제일 많이 먹고** → 강한 모델은 중간보다 덜 먹음.

SWE 기준 이득(Δbenefit):
- Qwen3-32B (base 3.6) → 4.4pp
- Qwen3-235B (base 20.7) → **19.3pp (peak)**
- GPT-OSS-120B (base 26.2) → 15.8pp
- Haiku 4.5 (base 66.0) → 2.4pp
- Opus 4.6 (base 74.2) → 2.6pp
(MCP에선 peak가 GPT-OSS-120B 7.0pp로 이동)

양 끝의 이유가 다른 게 포인트:
- 강한 쪽: ceiling effect. 이미 초기 하네스로 다 풀어서 올릴 여지가 없음 (납득됨)
- 약한 쪽: 천장으로 설명 **안 됨.** 개선 여지(headroom)가 제일 큰데도 이득이 제일 작음 → 진짜 흥미로운 지점

`<img — solver_side_delta_harness_swe_v2.pdf : Δbenefit vs base 그래프 (가운데 봉우리 모양)>`

==========================================
약한 쪽이 못 먹는 이유 — 실패 모드 2개

1. harness activation failure = "아예 안 불러옴"
skill을 식별은 하는데 context로 안 끌고 옴.
- skill-load rate(SLR): Qwen3-32B 0.251, GPT-OSS 0.446 vs 강한 모델 ~0.96
- 예(threejs): skill 로딩 요청을 별도 액션이 아니라 더 큰 액션 안에 끼워 넣음 → 환경이 유효한 load로 인식 못 함 → skill body가 끝내 context에 안 들어옴

2. harness adherence failure = "불러와도 안 따름"
- harness-following rate(HFR): Opus 0.757 vs Qwen3-32B 0.142
- 깔끔한 분리 사례 Qwen3-235B: SLR 0.961(Opus랑 거의 같음)인데 HFR은 0.350. → **불러오는 것만으로는 부족하다**
- long-horizon으로 갈수록 더 흘림(adherence drift): Qwen3-32B 로드 직후 0.52 → final 0.13 (-0.39 급락), Opus 0.89 → 0.80 (-0.09 안정)
- 예(pg-essay-to-audiobook): skill을 절차로 안 보고 완성된 스크립트로 취급. 첫 시도 실패하니까 처방된 fallback 안 밟고 그냥 포기

`<img — solver_side_failure.pdf : activation failure(왼쪽) / adherence failure(오른쪽) 두 케이스>`

==========================================
결론 — 설계 가이드 3개

1. **Capability budget은 evolver 말고 solver에 투자해라.**
   evolver 키워봤자 ≤3.1pp. 비용 빡빡하면 9B 오픈소스 evolver로 frontier evolver 대체 가능.

2. **harness invocation(불러오기)을 agent 학습에 1급 스킬로 박아넣어라.**
   약한 모델은 하네스를 아예 안 부른다 (SLR 25% vs 96%).

3. **long-horizon instruction following을 강화해라.**
   불러와도 trajectory 후반부로 갈수록 흘려버린다.

한 줄 정리: 하네스를 *만드는* 쪽은 모델 크기 무관(flat), *이득 보는* 쪽은 모델 능력에 강하게 의존하는데 그것도 비단조(non-monotonic). 그래서 돈은 solver에 써라.
